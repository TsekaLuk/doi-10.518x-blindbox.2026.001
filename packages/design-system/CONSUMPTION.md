# Design-System Consumption Boundaries

## Astryx Owns Components

Use Astryx for interactive and semantic UI primitives: `Button`, `TextInput`, `Banner`, `Text`, `Card`, `Dialog`, menus, form controls, status indicators, navigation primitives, and accessible states.

Import components through subpaths such as `@astryxdesign/core/Button`. Do not re-create Astryx controls with raw HTML plus Tailwind classes.

Astryx styling is driven by Astryx runtime tokens in `astryx.css`, for example `--color-background-body`, `--color-text-primary`, `--color-border`, and `--font-family-body`.

## Tailwind Owns App Layout

Use Tailwind for app shell layout, absolute positioning, responsive sizing, scene overlays, canvas chrome, spacing between app regions, and brand-only utility classes generated by `tailwind.css`.

Tailwind must not theme Astryx component internals. Avoid targeting Astryx generated classes or component markup. If an Astryx component needs a different visual system, map brand tokens into Astryx tokens in `astryx.css`.

## Brand Tokens Stay Neutral

`variables.css` defines project brand values under `--vibe-*` only. These are raw brand facts, not component styling rules.

`tailwind.css` exposes brand utilities for Tailwind only.

`astryx.css` maps brand values into Astryx component tokens only.

`DESIGN.md` is a reference document for humans and agents. Do not copy its raw Quick Start CSS into runtime files.
